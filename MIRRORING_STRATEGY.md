# 镜像翻转思路说明

这份文档只讲一件事：**如果要把左舵国家数据等效成右舵国家风格数据，最稳妥、最轻量的做法是什么。**

配合当前项目 `bev_debug_viewer`，建议把“镜像”理解为一个 **sample 级别的统一变换**，而不是一开始就重建一整套离线数据资产。

---

## 1. 核心结论

不要先离线重写整套 **pose / 地图 / 标定 / 图像资产**。

更推荐的做法是：

- 原始数据保持不变
- 在 dataloader 或 sample 构造阶段，在线生成一个 **mirrored sample**
- 网络或可视化只消费这份镜像后的 sample

也就是构造一个 **virtual mirrored rig**：

- 图像是镜像后的图像
- 相机几何是镜像后的几何
- LiDAR / bbox / 地图 / ego-frame target 也同步镜像
- 原始全局 pose、原始地图数据库、原始标定文件本身不改

这个思路的好处是：

- 软件改动最小
- 原始资产不被污染
- 最容易做原始/镜像的 A/B 对照
- 出 bug 时最好排查
- 真正耗时的部分可以后面再单独离线化

---

## 2. 要分清楚两类转换

### 第一类：偏原始传感器的量

这类量和当前 sample 的观测直接相关，通常是：

- 相机图像
- LiDAR 点云
- 当前 sample 的 bbox / 动标 GT
- 当前 sample 的局部地图表达

这类量的特点是：

- 和当前帧强相关
- 在线处理通常很直接
- 数学关系清楚，容易可视化验收

其中：

#### 图像
图像如果要做真正的“左舵变右舵”，**水平镜像通常是前提**。

因为对于 camera-to-BEV 网络，模型输入的视觉内容必须和几何变换后的相机坐标系保持一致。只改相机矩阵、不改图像，视觉和几何会不自洽。

图像镜像的缺点是有一定开销，所以它往往是最值得后续 profile 的一项。

#### LiDAR
LiDAR 通常比图像简单很多。点云如果已经在 ego frame 下，只需要做：

- `(x, y, z) -> (x, -y, z)`

就可以完成左右镜像。

它不一定需要离线预生成新文件；很多时候在加载后在线改一下就够了。这部分实现简单，也比较容易单独缓存。

---

### 第二类：偏全局几何和系统配置的量

这类量通常是：

- 原始全局 pose
- 原始地图数据库
- 原始相机/激光外参文件
- 原始标定资产

这类量的特点是：

- 链路长
- 依赖多
- 一旦离线改动，容易牵一发动全身
- 非常容易因为坐标系定义不统一而出错

所以一开始**不建议直接重写这些原始资产**。

更好的方式是：

- 保持原始 pose / 地图 / 标定不变
- 在 sample 构造时，统一叠加一个镜像操作
- 让所有真正进入网络或监督的量，在局部 sample 内部完成变换

这就是“在线构造 virtual mirrored rig”的核心思想。

---

## 3. 为什么 LSS 场景下不能只用现成的 rand_flip

这一点很关键。

在 LSS 这类 camera-to-BEV 架构里，代码里常见的 image flip 往往只是**图像增广**，不是“把左舵场景变成右舵场景”。

原因是：

- 图像 flip 会被记录到 `post_rot / post_tran`
- 后续几何投影时又会把这组 image-space 变换逆掉

所以这种 flip 的本质是：

- 保持同一个世界
- 只做图像域的数据增广

而不是：

- 构造一个真正镜像后的交通场景

因此，如果目标是“用左舵数据模拟右舵国家”，不能只打开官方已有的随机翻转开关。

必须单独有一个 **virtual_rhd / mirrored sample** 分支，把视觉、几何、监督一起同步改掉。

---

## 4. 在线镜像时，哪些量需要改

如果按 sample 在线构造镜像数据，建议统一按下面的规则处理。

### 4.1 图像
- 水平镜像

### 4.2 相机外参
如果点是按

- `p_ego = R * p_cam + t`

投到 ego 系，那么可用：

- `S_ego = diag(1, -1, 1)`
- `S_cam = diag(-1, 1, 1)`

构造镜像后的相机：

- `t' = S_ego @ t`
- `R' = S_ego @ R @ S_cam`

这一步的意义是：图像翻了，相机也要变成一个**自洽的虚拟镜像相机**。

### 4.3 LiDAR 点
如果点已经在 ego frame：

- `(x, y, z) -> (x, -y, z)`

### 4.4 bbox / 动标 GT
- `center_y -> -center_y`
- `yaw -> -yaw`
- 如果有速度/轨迹，`vy -> -vy`

### 4.5 地图 / 局部 raster / ego-frame target
只要它们最终是 **ego-frame 输入或监督**，就统一做：

- `y -> -y`

如果已经是 BEV raster / occupancy / mask，那么直接翻 lateral 维即可。

---

## 5. 最重要的判断：不是所有东西都需要离线翻一份

更直白地说：

### 不建议一开始就改的
- 原始地图数据库
- 原始全局 pose 文件
- 原始传感器标定文件
- 原始离线图像资产全集

### 建议先在线改的
- 当前 sample 的图像
- 当前 sample 的相机外参
- 当前 sample 的 LiDAR 点
- 当前 sample 的 bbox / 动标 GT
- 当前 sample 的 ego-frame 地图 / 栅格监督

也就是说，真正需要镜像的是：

> **那些真的作为模型输入或监督进入网络的量。**

而不是把整个数据系统从底层重做一遍。

---

## 6. 为什么建议先在线做，再把热点离线化

这是最推荐的工程推进方式。

### 第一步：先在线镜像
目的不是立刻提速，而是先确认：

- 数学关系对不对
- 视觉和几何是否自洽
- 图像、LiDAR、bbox、地图、frustum 是否都能对齐

当前项目 `bev_debug_viewer` 就是为这个目的服务的。它提供了统一的场景接口，可以把：

- 图像
- LiDAR
- bbox
- 地图
- frustum

一起画出来，做原始/镜像的 sanity check。

### 第二步：对在线链路做 profile
确认镜像逻辑正确后，再看真正耗时的是哪一段。

通常最值得看的有两类：

- 图像解码 / resize / flip
- 复杂地图栅格化或重投影

而 LiDAR 点的镜像、bbox 的镜像、yaw 取负这类操作，通常都不是瓶颈。

### 第三步：只把热点步骤离线缓存
当你已经知道哪一步慢，再把那一步做成离线预存。

例如：

- 预存镜像后的图像
- 预存固定分辨率图像 tensor
- 预存地图 raster
- 预存某些重复计算的局部监督

这样做的原则是：

- 先保证正确
- 再做缓存
- 只缓存真正贵的部分
- 不要把所有东西都提前固化

---

## 7. 对不同架构，处理方式不同

### 7.1 camera-to-BEV（例如 LSS 这一类）
这种情况下必须同时保证：

- 图像侧正确
- 相机几何侧正确
- ego-frame supervision 正确

所以图像和相机外参都要同步改，不能只翻 BEV 标签。

### 7.2 输入已经是现成的 BEV feature / occupancy / raster
这种情况下通常更简单。

因为相机几何链路已经不在当前网络里了，所以很多时候只需要把：

- BEV 输入
- BEV 标签
- ego-frame 辅助量

一起做左右翻转即可。

也就是说：

- 是否需要碰图像和相机外参，取决于网络输入是不是 raw image。

这是架构差异，不是数据本身的差异。

---

## 8. 两个必须提醒的坑

### 坑 1：带左右语义的标签不能只做几何镜像
如果任务里有这类显式语义：

- left-turn / right-turn
- 左变道 / 右变道
- 左侧路缘 / 右侧路缘
- 左优先 / 右优先

那么镜像之后还需要做 **label remap**。

否则几何翻对了，语义却反了。

### 坑 2：不要把“普通图像增广 flip”误当成“右舵模拟”
普通 flip 常常只是在图像域做增广，不会改变底层世界几何。

如果目标是 domain conversion，必须显式构造一个镜像后的 sample。

---

## 9. 这份思路如何对应当前项目代码

当前 `bev_debug_viewer` 项目已经提供了一个很适合做这件事的基础接口：

- `SceneRecord`
- `CameraRecord`
- `LidarRecord`
- `BoxRecord`
- `MapRecord`

推荐后续把镜像逻辑统一收口成：

```python
scene_m = mirror_scene(scene)
```

或者：

```python
scene_m = scene.mirrored()
```

这个镜像入口内部负责：

- 图像镜像
- 相机外参镜像
- LiDAR 点镜像
- bbox 镜像
- 地图镜像

然后再复用当前项目已有的可视化函数，把：

- 图像
- LiDAR
- frustum
- GT
- 地图

全部画出来，对比原始和镜像结果。

这一步跑通之后，再考虑接入真实 dataloader 或训练链路，顺序会稳很多。

---

## 10. 一句话总结

这件事最好的落地方式不是：

> 先把整套原始数据资产离线翻一遍。

而是：

> **先在 dataloader / sample 层在线构造一个 virtual mirrored rig，先验证数学和可视化都正确；再把真正耗时的局部步骤离线缓存。**

这样改动最小、风险最低，也最符合 LSS 这类 camera-to-BEV 系统的输入输出结构。
